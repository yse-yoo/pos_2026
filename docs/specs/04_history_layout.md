# 売上履歴画面 仕様書

**バージョン:** 1.0.0  
**作成日:** 2026-04-24  
**対象画面:** 売上履歴画面 (`/sales/history`)

---

## 1. 概要

売上履歴画面は、過去の会計データを時系列で一覧表示し、各売上の詳細画面へ遷移するための画面である。  
`backend/pos_2026/docs/screen_design.md` の「売上履歴画面」と `system_design.md` の SPA + API 構成、および `GET /api/sales` / `GET /api/sales/:id` を前提に設計する。

本画面の責務は **過去の売上一覧の確認と詳細画面への導線提供** に限定する。  
日別・月別集計や高度な検索は売上管理画面 (`/sales`) の責務とし、この画面では扱わない。

---

## 2. 画面の役割

| 項目 | 内容 |
|------|------|
| 画面名 | 売上履歴画面 |
| URL | `/sales/history` |
| 主な役割 | 過去の売上一覧を確認する |
| 遷移元 | レジ画面 (`/pos`)、管理者メニュー画面 (`/admin`) |
| 遷移先 | 売上詳細画面 (`/sales/:id`) |
| 利用API | `GET /api/sales` |

---

## 4. UI仕様

### 4.1 共通ヘッダー

既存のレジ画面と同じ共通ヘッダーを利用する。

| 要素 | 内容 |
|------|------|
| ロゴ | アプリ名またはブランド表示 |
| ナビゲーション | 「レジ」「履歴」「商品管理」など |
| 現在時刻 | 既存実装と同じリアルタイム表示 |
| 管理者導線 | 管理画面または管理者メニューへの導線 |

### 4.2 履歴パネルヘッダー

| 要素 | 内容 |
|------|------|
| 画面タイトル | `売上履歴` |
| 補足テキスト | 「過去の会計一覧を確認できます」などの短い説明 |
| 再読み込み | 一覧の再取得を行うボタン |

### 4.3 売上一覧

一覧は新しい会計が先頭に来る降順表示を基本とする。

| 列 | 内容 |
|----|------|
| 会計日時 | 会計完了日時 |
| 伝票番号 | レジ画面で採番される伝票番号 |
| 注文点数 | 明細の合計点数 |
| 合計金額 | 税込合計金額 |
| 支払方法 | 現金 / カード など |
| 操作 | `詳細を見る` ボタンまたは行全体クリック |

- 1行が1件の売上を表す
- 金額は `¥1,234` 形式で表示する
- 伝票番号は `NO.0001` のようなゼロ埋め形式を維持する
- 詳細操作で売上詳細画面 (`/sales/:id`) へ遷移する

### 4.4 空状態

履歴が0件の場合は空状態を表示する。

| 要素 | 内容 |
|------|------|
| アイコン | 履歴を示す簡易アイコン |
| 見出し | `会計履歴` |
| メッセージ | `履歴データはまだありません` |

### 4.5 読み込み状態

- 初回表示時はローディング表示を行う
- 再読み込み中は一覧を維持したままローディング表示を重ねるか、ボタンを無効化する

### 4.6 エラー状態

- API取得失敗時は、履歴取得に失敗した旨を画面内に明示する
- 再試行導線として `再読み込み` を表示する

---

## 5. 画面遷移仕様

| 操作 | 遷移先 / 動作 |
|------|---------------|
| レジ画面から「履歴」を選択 | `/sales/history` を表示 |
| 管理者メニューから履歴を選択 | `/sales/history` を表示 |
| 一覧の「詳細を見る」を押下 | `/sales/:id` へ遷移 |
| 再読み込みを押下 | `GET /api/sales` を再実行 |

---

## 6. API連携仕様

### 6.1 一覧取得

| 項目 | 内容 |
|------|------|
| メソッド | `GET` |
| エンドポイント | `/api/sales` |
| 用途 | 売上履歴一覧の取得 |
| 取得タイミング | 画面初回表示時、再読み込み時 |

### 6.2 詳細遷移

| 項目 | 内容 |
|------|------|
| メソッド | `GET` |
| エンドポイント | `/api/sales/:id` |
| 用途 | 売上詳細画面で1件の売上詳細を取得 |

### 6.3 フロントエンドで必要な一覧データ

`GET /api/sales` のレスポンスには少なくとも以下の項目が必要である。

| フィールド | 型 | 説明 |
|-----------|----|------|
| `id` | number | 売上ID |
| `receiptNumber` | string | 伝票番号 |
| `soldAt` | string | 会計日時 |
| `itemCount` | number | 注文点数 |
| `totalAmount` | number | 税込合計金額 |
| `paymentMethod` | string | 支払方法 |

レスポンス例:

```json
[
  {
    "id": 101,
    "receiptNumber": "NO.0001",
    "soldAt": "2026-04-24T14:32:00+09:00",
    "itemCount": 3,
    "totalAmount": 1850,
    "paymentMethod": "現金"
  }
]
```

## 7. 状態管理仕様

| 状態 | 型 | 説明 |
|------|----|------|
| `salesHistory` | Array | 一覧表示用の売上データ |
| `isLoading` | boolean | 初回読込中フラグ |
| `isRefreshing` | boolean | 再取得中フラグ |
| `errorMessage` | string \| null | エラーメッセージ |

- 初期表示時に `isLoading = true`
- 取得成功時は `salesHistory` を更新し `isLoading = false`
- 取得失敗時は `errorMessage` を設定する

## 8. デザイン方針

`frontend/docs/mock/03_fix_layout.md` と既存フロントの配色方針に合わせ、以下を基本とする。

- エコなグリーン系をベースカラーに使う
- 文字サイズはやや大きめで視認性を優先する
- 背景は淡色、情報領域は白カードで分離する
- 重要操作は主ボタン、再読み込みなど補助操作は副ボタンで区別する
- 履歴画面でも既存ヘッダーとナビゲーションの見た目を統一する

---

## 9. レスポンシブ方針

- **PC:** 表形式またはカード一覧で主要情報を横並び表示
- **タブレット:** 列数を絞り、操作列を右端に固定
- **スマートフォン:** 1件ごとの縦積みカード表示に切り替え、日時・金額・詳細導線を優先表示

---

## 10. 実装上の補足

- 本画面は SPA の一画面として実装する
- データ取得は `fetch()` を用いる
- API通信結果は JSON とする
- 将来的に React Router を導入する場合でも、画面URLは `screen_design.md` に合わせて `/sales/history` を基準とする
- 集計・検索要件が追加された場合は、履歴画面に肥大化させず売上管理画面へ切り分ける
